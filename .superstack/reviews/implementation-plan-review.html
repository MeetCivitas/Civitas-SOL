Civitas-Sol Implementation Plan Review

Scope: review of implementation_plan.md.resolved using the Solana review rubric with emphasis on security, correctness, compute, and production readiness.

Score

Security: C
Quality: B
Ready for mainnet: No

The plan is strong on product vision and privacy layering. The main gaps are Solana execution realism: compute viability for UltraHonk, batching and state layout, proof-domain binding, and Token-2022 confidential account lifecycle.

Findings

HighUltraHonk verifier is the entire critical path. The plan assumes a custom split-transaction verifier will fit Solana constraints, but does not include a fallback if CU, memory, or engineering complexity make that unworkable in time.

Fix: add a fallback milestone for the hackathon demo, with the on-chain verifier as a stretch milestone rather than the only settlement path.

HighCommit batching is not Solana-native yet. A single update_merkle_root(root, commitments) path is too optimistic once payroll runs grow beyond a tiny batch.

Fix: redesign around chunked append/finalize instructions and budget for versioned transactions plus lookup tables.

HighProof replay boundaries are underspecified. Binding only a recipient hash and token account hash is not enough for Solana deployment boundaries.

Fix: bind the proof to program_id, vault_pda, mint, recipient token account, run or epoch id, and a deployment-domain tag.

MediumToken-2022 confidential balances need their own workstream. The current plan treats them as a thin helper library instead of a key-management and account-provisioning subsystem.

Fix: add an explicit onboarding phase for confidential token accounts, decrypt keys, recovery, and auditor-key policy.

MediumRegistry account design is demo-safe but not scale-safe. One PDA per commitment and nullifier is operationally simple, but rent-heavy and write-heavy.

Fix: document the demo approach and the scale path separately, using chunked registries, bitmaps, or append-only epoch accounts.

MediumThe verification plan is missing Solana failure testing. The plan lists happy-path tests, but not compute-budget regressions, tx-size tests, local-validator confidential token tests, or finalized-state reconciliation.

Fix: add those tests before calling the plan production-ready.

What To Add

  • A fallback delivery plan with “demo-safe” and “full cryptographic” milestones.
  • A chunked payroll commit flow instead of a single giant commitment vector.
  • Explicit authority model: upgrade authority, pause authority, vault authority, and who can finalize payroll runs.
  • A proof-domain separator section covering replay protection across vaults, mints, accounts, clusters, and deployments.
  • A Token-2022 confidential balances lifecycle section covering setup, recovery, and UX constraints.
  • A Solana-specific test matrix for CU, transaction size, replays, bad roots, aliasing, and account validation.
